What's a clean way to structure error handling in this service?

The cleanest approach is to centralize your errors into typed classes and let a single boundary translate them into responses. That keeps route handlers readable and logging consistent.

  • One base AppError with a stable code and HTTP status.
  • Subclasses for the cases you actually branch on.
  • A single middleware that maps any thrown error to a response.

From there the client always gets a predictable shape it can branch on.

Should validation errors go through the same boundary?

Yes — wrap them as a ValidationError so they share the same envelope. Carry the field-level details in a details array, and your boundary returns 422 automatically. One code path, one response shape, no special-casing in handlers.
